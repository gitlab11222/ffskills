# 需求文档规范

## PRD文档结构

### 标准PRD模板
```markdown
# 产品需求文档：{功能名称}

## 1. 需求概述

### 1.1 业务背景
- **目标**: {业务目标描述}
- **用户**: {目标用户群体}
- **场景**: {使用场景描述}
- **痛点**: {当前存在的问题}

### 1.2 功能范围
- **包含**: {本次实现的功能列表}
- **不包含**: {明确不在本次范围的内容}
- **后续规划**: {未来可能的功能扩展}

### 1.3 优先级
| 功能 | 优先级 | 说明 |
|------|--------|------|
| {功能1} | P0 | 核心功能，必须实现 |
| {功能2} | P1 | 重要功能，建议实现 |
| {功能3} | P2 | 优化功能，后续迭代 |
```

```
## 3. 业务流程

### 3.1 主流程图
```
开始 → 步骤1 → 步骤2 → 步骤3 → 结束
                    ↓
                  异常处理
```

### 3.2 流程步骤表
| 步骤ID | 步骤名称 | 操作 | 触发条件 | 后续动作 | 异常处理 |
|--------|----------|------|----------|----------|----------|
| S001 | 创建 | 用户提交 | 用户点击提交 | 校验数据 | 数据校验失败提示 |
| S002 | 校验 | 系统校验 | 提交后执行 | 校验成功则保存 | 返回错误信息 |
| S003 | 保存 | 系统保存 | 校验成功 | 返回成功 | 保存失败回滚 |
| S004 | 完成 | 显示结果 | 保存成功 | 提示成功 | - |

### 3.3 状态流转图
```
待支付 → 已支付 → 待发货 → 已发货 → 已完成
   ↓        ↓        ↓
 已取消   已退款   已退货
```

### 3.4 状态定义表
| 状态ID | 状态值 | 标签 | 允许操作 | 后续状态 | 说明 |
|--------|--------|------|----------|----------|------|
| ST001 | 10 | 待支付 | 支付、取消 | 已支付、已取消 | 初始状态 |
| ST002 | 20 | 已支付 | 发货、退款 | 待发货、已退款 | 支付成功 |
| ST003 | 30 | 待发货 | 发货 | 已发货 | 待发货状态 |
| ST004 | 40 | 已发货 | 确认收货 | 已完成 | 已发货状态 |
| ST005 | 50 | 已完成 | 评价 | - | 订单完成 |
| ST006 | -10 | 已取消 | - | - | 订单取消 |
| ST007 | -20 | 已退款 | - | - | 订单退款 |

## 4. 业务规则

### 4.1 校验规则

#### 创建规则
- **R001**: 库存必须 ≥ 订单数量
- **R002**: 单次下单数量 ≤ 100
- **R003**: 订单金额 ≥ 0.01 元
- **R004**: 商品状态必须为"正常"

#### 更新规则
- **R005**: 仅可更新"待支付"状态订单
- **R006**: 更新后需校验数据有效性

#### 删除规则
- **R007**: 仅可删除"待支付"状态订单
- **R008**: 删除后需恢复库存

### 4.2 权限规则
| 角色 | 权限ID | 权限码 | 操作权限 | 数据范围 | 说明 |
|------|--------|--------|----------|----------|------|
| 管理员 | P001 | all | 全部 | 全部数据 | 可查看、编辑、删除所有数据 |
| 运营 | P002 | edit | 查看、编辑 | 自己负责的数据 | 可查看和编辑，不可删除 |
| 客服 | P003 | view | 查看 | 全部数据 | 仅可查看 |

### 4.3 异常处理规则
| 异常ID | 异常场景 | 处理方式 | 用户提示 | 系统动作 |
|--------|----------|----------|----------|----------|
| E001 | 库存不足 | 阻止创建 | "库存不足，请减少数量" | 返回错误，不创建订单 |
| E002 | 支付超时 | 自动取消 | "订单已超时取消" | 取消订单，恢复库存 |
| E003 | 支付失败 | 保持待支付 | "支付失败，请重试" | 记录日志，状态不变 |
```

## 需求描述规范

### 需求描述三要素

每个需求描述必须包含：

1. **背景**: 为什么需要这个功能
2. **内容**: 具体要实现什么
3. **价值**: 能带来什么收益

```markdown
// ✓ 正确示例
### 需求背景
当前用户下单后需要等待客服手动处理，效率低、体验差。

### 需求内容
实现用户自助查询订单状态，支持订单详情查看、物流跟踪、取消订单操作。

### 预期效果
- 用户自助处理率提升50%
- 客服工作量减少30%
- 用户满意度提升20%

// ✗ 错误示例
### 需求内容
做一个订单查询功能。  // 缺少背景和价值说明
```

### 功能描述规范

```markdown
// ✓ 正确示例
### 功能: 用户登录

**描述**: 用户通过手机号和密码登录系统，登录成功后获取Token，用于后续接口认证。

**输入**:
- 手机号: 11位数字，必填
- 密码: 6-20位字符，必填

**处理**:
1. 校验手机号格式
2. 校验密码格式
3. 查询用户是否存在
4. 验证密码是否匹配
5. 生成Token
6. 返回Token和用户信息

**输出**:
- 成功: Token、用户基本信息
- 失败: 错误信息

**异常**:
- 用户不存在: 提示"用户不存在"
- 密码错误: 提示"密码错误"
- 格式错误: 提示具体校验失败信息

// ✗ 错误示例
### 功能: 用户登录
实现登录功能。  // 描述不完整
```

## 需求评审规范

### 评审准备

| 准备项 | 要求 |
|--------|------|
| PRD文档 | 完整填写，无缺失字段 |
| 原型设计 | 至少有低保真原型 |
| 数据模型 | 表结构、字段定义清晰 |
| 验收标准 | 明确可测试 |

### 评审内容

| 评审维度 | 检查内容 |
|----------|----------|
| 业务价值 | 目标清晰、价值量化 |
| 功能范围 | 包含/不包含明确 |
| 技术可行性 | 技术方案可行 |
| 验收标准 | 标准明确可测试 |
| 风险评估 | 风险识别、应对方案 |
