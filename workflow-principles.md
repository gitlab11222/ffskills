# 研发工作流核心原则

## 确认节点规范

### 必须确认的节点
| 节点 | 确认内容 | 确认方式 | 确认人 |
|------|----------|----------|--------|
| 需求确认 | PRD文档、业务流程、验收标准 | 评审会议+签字 | 产品+研发+测试 |
| 原型确认 | 页面布局、交互流程、功能入口 | 评审会议+签字 | 产品+UX+前端 |
| UI确认(demo) | 视觉设计、颜色字体、组件样式 | 评审会议+签字 | 产品+UX+前端 |
| 数据库表设计确认 | 表结构、字段定义、索引设计 | 评审会议+签字 | 架构师+DBA+后端 |
| 技术架构确认 | 系统架构、技术选型、性能方案 | 评审会议+签字 | 架构师+技术经理 |
| 测试用例确认 | 用例覆盖、测试场景、验收标准 | 评审会议+签字 | 测试+研发+产品 |

### 确认节点执行顺序

**执行顺序：严格按以下顺序执行，禁止跳过前置步骤**

| 顺序 | 节点 | 前置条件 | 禁止事项 |
|------|------|----------|----------|
| 1 | 需求确认 | 无 | 直接进入原型设计 |
| 2 | 原型确认 | 需求确认签字完成 | 直接进入数据库设计 |
| 3 | UI确认(demo) | 原型确认签字完成 | 直接进入开发 |
| 4 | 数据库设计确认 | UI确认签字完成 | 直接进入代码开发 |
| 5 | 技术架构确认 | 数据库设计确认签字完成 | 直接进入开发 |
| 6 | 测试用例确认 | 技术架构确认签字完成 | 直接发布 |

```markdown
// ✗ 禁止跳过前置节点
需求确认 → 数据库设计 → 代码开发  // 禁止，跳过原型和UI确认

// ✓ 正确执行顺序
需求确认 → 原型确认 → UI确认 → 数据库设计确认 → 技术架构确认 → 测试用例确认 → 代码开发

// ✗ 用户习惯中的错误示范
原型确认签字完成 → 直接进入数据库设计 → 数据库确认签字完成 → 代码开发  // 禁止，跳过UI确认

// ✓ 用户习惯的正确做法
原型确认签字完成 → UI确认签字完成 → 数据库设计 → 数据库确认签字完成 → 代码开发
```

**前置条件说明：**
- **原型确认的前置条件**：需求文档评审通过且签字完成
- **UI确认的前置条件**：原型评审通过且签字完成
- **数据库设计确认的前置条件**：UI确认签字完成、原型设计定稿
- **技术架构确认的前置条件**：数据库设计评审通过且签字完成
- **测试用例确认的前置条件**：技术架构评审通过且签字完成
- **代码开发的前置条件**：上述所有确认节点全部签字完成

## 代码编写规则

### 前端代码编写规则
| 规则 | 说明 |
|------|------|
| 一次性写完 | 前端代码可以一次性写完，无需每一步都确认 |
| 功能完整 | 一个功能模块的代码应完整交付，不要分多次提交 |
| 自测验证 | 代码完成后必须自测验证，确保功能可用 |
| 统一提交 | 功能完成后统一提交代码审查 |

```markdown
// ✓ 前端一次性写完
一个页面/模块的代码一次性完成：
- 页面组件
- API接口调用
- 状态管理
- 表单校验
- 错误处理

完成后自测验证 → 提交代码审查

// ✗ 禁止分步确认
写一个组件就问一下  // 禁止，效率低

// ✓ 正确做法
完整功能一次性完成 → 自测 → 提交审查
```

### 后端代码编写规则
| 规则 | 说明 |
|------|------|
| 一次性写完 | 后端代码可以一次性写完，无需每一步都确认 |
| 功能完整 | 一个接口/服务的代码应完整交付，包含实体、DTO、服务、Controller |
| 自测验证 | 代码完成后必须自测验证，确保接口可用 |
| 统一提交 | 功能完成后统一提交代码审查 |

```markdown
// ✓ 后端一次性写完
一个接口的代码一次性完成：
- 实体定义
- DTO定义
- 服务实现
- Controller接口
- 参数校验
- 异常处理
- 日志记录

完成后自测验证 → 提交代码审查

// ✗ 禁止分步确认
写一个实体就问一下  // 禁止，效率低

// ✓ 正确做法
完整接口一次性完成 → 自测 → 提交审查
```

### 代码编写确认点
| 确认点 | 说明 | 确认时机 |
|--------|------|----------|
| 技术方案确认 | 编写前确认技术方案 | 开发前 |
| 代码审查 | 代码完成后审查 | 开发后 |
| 功能验收 | 功能完成后验收 | 测试后 |

## 问题处理规则

### 数据库连接问题处理
| 规则 | 说明 |
|------|------|
| 不私自做决定 | 数据库连接不上等问题，不要私自做决定 |
| 不私自换数据库 | 禁止私自采用SQLite等替代数据库 |
| 及时询问 | 遇到问题不动就问，询问相关负责人 |
| 记录问题 | 记录问题详情，便于排查和解决 |

```markdown
// ✗ 禁止私自换数据库
数据库连不上MySQL → 私自换成SQLite  // 禁止

// ✓ 正确做法
数据库连接失败 → 记录错误信息 → 询问DBA/运维 → 等待解决

// ✗ 禁止私自修改配置
连接超时 → 私自修改超时时间为1秒  // 禁止

// ✓ 正确做法
连接超时 → 记录超时时间 → 询问负责人 → 按建议处理
```

### 问题询问流程
```
发现问题 → 记录详情 → 确定负责人 → 询问解决方案 → 执行解决 → 记录结果
```

### 常见问题处理
| 问题类型 | 处理方式 | 询问对象 |
|----------|----------|----------|
| 数据库连接失败 | 记录错误信息，询问DBA | DBA/运维 |
| Redis连接失败 | 记录错误信息，询问运维 | 运维 |
| 第三方接口异常 | 记录响应信息，询问架构师 | 架构师 |
| 配置参数缺失 | 记录缺失项，询问技术经理 | 技术经理 |
| 权限不足 | 记录权限需求，询问管理员 | 系统管理员 |

## 架构简化原则

### 架构选择原则
| 原则 | 说明 |
|------|------|
| 高级全栈工程师视角 | 以实际开发维护视角设计架构 |
| 代码可维护性优先 | AI写的代码需要人来维护，必须简单易懂 |
| 拒绝不必要复杂度 | 不要每个项目都DDD、聚合根 |
| 轻量架构优先 | 优先选择简单直接的架构方案 |

### 架构选择对照
| 项目类型 | 推荐架构 | 禁止架构 | 说明 |
|----------|----------|----------|------|
| 简单CRUD | 分层架构（Controller-Service-Repository） | DDD、聚合根 | 简单项目不需要DDD |
| 中型项目 | 分层架构 + 部分领域划分 | 全DDD、聚合根 | 按需引入领域概念 |
| 复杂系统 | 部分DDD（核心领域） | 强制全部DDD | 核心领域才用DDD |

```markdown
// ✓ 简单CRUD项目用分层架构
Controller → Service → Repository → Entity
简单清晰，易于理解和维护

// ✗ 禁止简单项目用DDD
简单订单管理 → DDD + 聚合根 + 领域事件 + 值对象  // 禁止，过度设计

// ✓ 正确做法
简单订单管理 → Controller → OrderService → OrderRepository → Order

// ✓ 复杂核心领域才用DDD
支付系统 → 领域划分 → 聚合根（订单聚合） → 领域服务
复杂业务逻辑才需要DDD

// ✗ 禁止强制全部DDD
用户管理 → DDD聚合根  // 禁止，用户管理很简单

// ✓ 正确做法
用户管理 → UserService → UserRepository → User
```

### 架构复杂度评估
| 评估项 | 简单项目 | 中型项目 | 复杂项目 |
|--------|----------|----------|----------|
| 业务逻辑复杂度 | ≤ 10条规则 | 10-50条规则 | > 50条规则 |
| 数据表数量 | ≤ 5张表 | 5-20张表 | > 20张表 |
| 业务流程数量 | ≤ 3个流程 | 3-10个流程 | > 10个流程 |
| 是否需要DDD | ❌ 不需要 | ⚠️ 按需引入 | ✅ 核心领域需要 |

## 最佳实践

### 确认流程规范化
- 关键节点必须确认签字
- 确认记录存档保存
- 确认后方可进入下一阶段
