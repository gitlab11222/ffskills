# WBS生成 Skill

根据需求文档和团队成员，生成工作分解结构（WBS）和研发排期计划表。

## 使用方式

```
/create-wbs <需求ID>
/create-wbs --from-prd <需求文档路径>
/create-wbs --team <团队成员列表>
```

## 参数说明

| 参数 | 格式 | 说明 | 示例 |
|------|------|------|------|
| 需求ID | 数字 | TAPD需求ID | 12345 |
| --from-prd | 文件路径 | PRD文档路径 | docs/prd.md |
| --team | 成员列表 | 团队成员配置 | 张三:前端,李四:后端 |

## 选项

- `--duration N` - 预计开发周期（人天）
- `--buffer N` - 缓冲时间比例（默认10%）
- `--milestones` - 自定义里程碑节点

## 执行步骤

### 1. 分析需求范围

#### 1.1 功能模块拆分
- 识别功能模块
- 列出子功能清单
- 确定模块依赖关系

#### 1.2 任务类型分类
| 类型 | 说明 | 产出 |
|------|------|------|
| 前端任务 | 页面开发、组件开发 | 页面、组件代码 |
| 后端任务 | API开发、服务开发 | API代码、服务代码 |
| 数据库任务 | 表设计、迁移脚本 | SQL脚本 |
| 测试任务 | 用例编写、测试执行 | 测试用例、测试报告 |
| 设计任务 | 原型设计、UI设计 | 设计稿 |

### 2. 工时估算

#### 2.1 估算方法
- 功能复杂度评估
- 技术难度评估
- 团队成员能力系数
- 缓冲时间（10-20%）

#### 2.2 工时参考表
| 任务类型 | 简单 | 中等 | 复杂 |
|----------|------|------|------|
| 列表页开发 | 1-2人天 | 2-3人天 | 3-5人天 |
| 详情页开发 | 0.5-1人天 | 1-2人天 | 2-3人天 |
| 表单页开发 | 1-2人天 | 2-3人天 | 3-5人天 |
| API接口开发 | 0.5-1人天 | 1-2人天 | 2-3人天 |
| 数据表设计 | 0.5人天 | 1人天 | 1-2人天 |
| 测试用例编写 | 0.5人天/模块 | 1人天/模块 | 2人天/模块 |
| 原型设计 | 0.5人天 | 1人天 | 2人天 |

### 3. 任务分配

#### 3.1 分配原则
- 人岗匹配（技能匹配）
- 均匀分配（避免过载）
- 依赖协调（前置任务先行）

#### 3.2 团队成员配置
```markdown
## 团队成员配置

| 角色 | 姓名 | 投入比例 | 技能标签 |
|------|------|----------|----------|
| 前端 | 张三 | 100% | React, AntDesign |
| 后端 | 李四 | 50% | .NET, EFCore |
| 后端 | 王五 | 100% | .NET, Redis |
| 测试 | 马六 | 100% | 功能测试 |
| 设计 | 陈七 | 50% | 原型设计 |
```

### 4. 里程碑设置

#### 4.1 标准里程碑
| 里程碑 | 说明 | 时间点 |
|--------|------|--------|
| 原型评审 | 原型定稿 | 开发开始前 |
| 开发启动 | 开始开发 | 原型评审后 |
| 转测试 | 研发自测完成 | 开发完成后 |
| 产品验收 | 预发布验收 | 测试通过后 |
| 发布上线 | 正式发布 | 验收通过后 |

### 5. 输出WBS和排期表

## 示例

```
/create-wbs --from-prd docs/prd.md --team 张三:前端,李四:后端,马六:测试
```

生成结果包含：
- WBS分解（按角色分组）
- 工时汇总（含缓冲）
- 里程碑时间表
- 详细排期表（周/日）
- 资源安排表

## 注意事项

1. 工时估算需参考团队成员能力
2. 缓冲时间建议预留10-20%
3. 依赖关系需明确标注
4. 风险预案需提前准备
5. 每日站会跟踪进度
