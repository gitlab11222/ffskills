# 测试规范

## 测试流程

### 测试阶段
```
需求评审 → 测试计划 → 测试用例 → 测试执行 → Bug修复 → 回归测试 → 测试报告
```

| 阶段 | 输入 | 输出 | 责任人 |
|------|------|------|--------|
| 需求评审 | PRD、原型 | 测试要点、疑问 | 测试 |
| 测试计划 | 需求范围 | 测试计划文档 | 测试 |
| 测试用例 | 测试计划 | 测试用例文档 | 测试 |
| 测试执行 | 测试用例 | Bug记录、执行记录 | 测试 |
| Bug修复 | Bug详情 | 修复代码 | 研发 |
| 回归测试 | 修复验证 | 回归报告 | 测试 |
| 测试报告 | 测试结果 | 测试报告 | 测试 |

## 测试类型

### 功能测试

#### 测试范围
- **正常流程**: 按业务流程正常执行
- **异常流程**: 异常场景、错误处理
- **边界值**: 数值边界、长度边界、时间边界
- **兼容性**: 浏览器兼容、移动端适配

### 验证点

### 接口测试

#### 接口测试范围
- **参数校验**: 必填、格式、范围
- **业务逻辑**: 业务规则、状态流转
- **响应结构**: code、message、data
- **异常处理**: 业务异常、系统异常

### 安全测试

#### 安全测试清单

| 测试项 | 测试方法 | 预期结果 |
|--------|----------|----------|
| SQL注入 | 输入 `' OR 1=1--` | 返回参数错误，不执行注入 |
| XSS注入 | 输入 `<script>alert(1)</script>` | 内容被转义或过滤 |
| CSRF攻击 | 跨站请求伪造 | 验证Token有效 |
| 越权访问 | 用用户A访问用户B数据 | 返回无权限 |
| 未授权访问 | 无Token访问需授权接口 | 返回未登录 |
| 参数篡改 | 修改关键参数值 | 验证失败或无权限 |

#### SQL注入测试用例
```markdown
## SEC-SQL-001: SQL注入测试

### 测试场景
- 查询参数注入
- 排序参数注入
- 过滤条件注入

### 测试数据
| 输入 | 说明 |
|------|------|
| `' OR 1=1--` | 基础注入 |
| `; DROP TABLE users--` | 删除表注入 |
| `' UNION SELECT * FROM users--` | 联合查询注入 |
| `1 AND 1=1` | 条件注入 |

### 验证方式
- 响应不包含异常数据
- 响应返回参数错误
- 数据库无异常执行记录
```

## 测试用例编写规范

### 用例编号规则
```
TC-{模块}-{编号}
API-{模块}-{编号}
SEC-{类型}-{编号}
PER-{模块}-{编号}
```

| 类型 | 编号格式 | 说明 |
|------|----------|------|
| 功能测试 | TC-{模块}-{编号} | 如 TC-ORDER-001 |
| 接口测试 | API-{模块}-{编号} | 如 API-USER-001 |
| 安全测试 | SEC-{类型}-{编号} | 如 SEC-SQL-001 |
| 性能测试 | PER-{模块}-{编号} | 如 PER-ORDER-001 |

### 用例优先级定义

| 优先级 | 定义 | 覆盖要求 |
|--------|------|----------|
| P0 | 核心功能、高风险 | 100%覆盖，每次回归必测 |
| P1 | 重要功能、中风险 | 100%覆盖，版本发布前测试 |
| P2 | 一般功能、低风险 | 80%覆盖，按需测试 |

### 用例设计方法

#### 等价类划分
```
有效等价类: 符合规则的输入集合
无效等价类: 不符合规则的输入集合

示例:
- 有效等价类: 1-100 的数量
- 无效等价类: 0, 101, -1, abc
```

#### 边界值分析
```
边界值: 最小值、最小值-1、最大值、最大值+1

示例:
- 数量边界: 1(最小), 0(无效), 100(最大), 101(无效)
- 字符长度: 1, 0, 200, 201
```

#### 错误推测
```
根据经验推测可能出错的情况

常见错误场景:
- 参数缺失
- 参数类型错误
- 参数超长
- 特殊字符
- 并发操作
- 依赖数据不存在
```

#### 因果图法
```
分析输入条件与输出结果的关系

示例:
- 条件: 库存充足 + 用户未下单
- 结果: 允许下单
```

## Bug 管理规范

### Bug 分级标准

| 级别 | 定义 | 示例 | 处理时限 |
|------|------|------|----------|
| 🔴 P0 | 系统崩溃、核心功能不可用 | 无法登录、支付失败 | 立即修复 |
| 🟡 P1 | 重要功能异常、有规避方案 | 搜索无结果、排序错误 | 24小时内 |
| 🟢 P2 | 一般问题、体验优化 | UI错位、提示不清 | 发布前修复 |

### Bug 状态流转
```
新建 → 分配 → 处理中 → 待验证 → 已关闭
                ↓
            无法修复 → 挂起
                ↓
            重复 → 已关闭
```

| 状态 | 说明 | 操作人 |
|------|------|--------|
| 新建 | Bug刚发现 | 测试 |
| 分配 | 分配给开发 | 测试 |
| 处理中 | 开发正在修复 | 开发 |
| 待验证 | 修复完成待验证 | 开发 |
| 已关闭 | 验证通过 | 测试 |
| 挂起 | 暂不处理 | 开发 |
| 重复 | 已有相同Bug | 测试 |

## 回归测试规范

### 回归测试时机
- 每次版本发布前
- Bug 修复后
- 重大功能变更后

### 回归测试范围

| 优先级 | 覆盖范围 | 说明 |
|--------|----------|------|

## 最佳实践

### 测试数据准备
1. 提前准备测试数据，覆盖各种场景
2. 测试数据与生产数据隔离
3. 预发布环境使用生产镜像数据

### Bug 描述规范
1. 描述具体现象，而非主观判断
2. 复现步骤可重现
3. 提供截图、日志等证据

### 测试报告规范
1. 数据量化，有具体数字
2. 结论明确，有具体建议
3. 遗留问题有规避方案和后续计划
