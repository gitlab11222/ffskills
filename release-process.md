# 发布流程规范

## 发布阶段

### 发布流程概览
```
开发完成 → 研发自测 → 转测试 → 测试验收 → 产品验收 → 发布上线 → 上线验证
```

| 阶段 | 输入 | 输出 | 负责人 |
|------|------|------|--------|
| 开发完成 | 需求文档 | 功能代码 | 研发 |
| 研发自测 | 功能代码 | 自测报告、转测试单 | 研发 |
| 测试验收 | 转测试单 | 测试报告 | 测试 |
| 产品验收 | 测试报告 | 验收确认 | 产品 |
| 发布上线 | 验收确认 | 发布报告 | 运维 |
| 上线验证 | 发布报告 | 上线确认 | 运维/运营 |

## 回滚流程

### 回滚触发条件
| 条件 | 触发级别 | 处理方式 |
|------|----------|----------|
| 服务启动失败 | 🔴 立即回滚 | 执行回滚脚本 |
| 核心功能不可用 | 🔴 立即回滚 | 执行回滚脚本 |
| 数据迁移失败 | 🔴 立即回滚 | 恢复数据备份 |
| 性能严重下降 | 🟡 评估回滚 | 分析原因后决定 |
| 非核心功能异常 | 🟢 修复上线 | 发布修复版本 |

### 回滚步骤
```
1. 停止服务 → 2. 恢复数据备份 → 3. 部署旧版本 → 4. 验证功能 → 5. 发送回滚通知
```

### 回滚脚本模板
```bash
#!/bin/bash
# 回滚脚本：{项目}-{版本}

# 1. 停止服务
echo "停止服务..."
docker stop {service_name}
docker rm {service_name}

# 2. 恢复数据备份（如有数据库变更）
echo "恢复数据备份..."
mysql -h{host} -u{user} -p{password} {database} < {backup_file}

# 3. 部署旧版本
echo "部署旧版本..."
docker run -d --name {service_name} {image}:{old_version}

# 4. 验证服务
echo "验证服务..."
sleep 10
curl -s http://localhost/hc

echo "回滚完成"
```

## 灰度发布

### 灰度发布策略
| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 按用户分批 | 按用户ID/地区逐步放量 | 大版本发布 |
| 按功能开关 | 新功能对部分用户开放 | 功能试运行 |
| AB测试 | 两组用户对比验证 | 优化验证 |

### 灰度发布步骤
```
1. 部署新版本到灰度环境
2. 配置灰度规则（用户/地区）
3. 灰度用户验证
4. 监控灰度指标
5. 逐步扩大灰度范围
6. 全量发布
```

### 灰度配置示例
```yaml
# 灰度发布配置
grayscale:
  strategy: user_id  # 按用户ID
  ratio: 10%         # 灰度比例
  rules:
    - user_id % 10 == 0  # 10%用户
  monitoring:
    metrics: [error_rate, response_time, conversion]
    thresholds:
      error_rate: 1%
      response_time: 500ms
```

## 发布版本管理

### 版本号规则
```
{主版本}.{次版本}.{修订号}

示例：
- V1.0.0：首次发布
- V1.1.0：新增功能
- V1.0.1：Bug修复
```

| 版本类型 | 说明 | 示例 |
|----------|------|------|
| 主版本 | 重大变更、架构调整 | V2.0.0 |
| 次版本 | 新增功能、功能优化 | V1.1.0 |
| 修订号 | Bug修复、小调整 | V1.0.1 |

### 发布分支策略
| 分支 | 说明 | 合并规则 |
|------|------|----------|
| develop | 开发分支 | 功能分支合并至此 |
| release/* | 发布分支 | develop 合并至此 |
| master | 生产分支 | release 合并至此 |
| hotfix/* | 紧急修复 | 直接合并至 master |

```markdown
// ✗ 发布前未测试
直接发布到生产环境  // 禁止

// ✓ 正确做法
测试通过 → 验收通过 → 发布

// ✗ 发布无监控
发布后无人监控  // 禁止

// ✓ 正确做法
发布过程有人监控，发布后观察30分钟

// ✗ 发布后无通知

// ✓ 正确做法
选择低峰时段发布（10:00-18:00）
```

## 最佳实践

### 发布时间选择
1. 选择工作日低峰时段（10:00-18:00）
2. 避开周五下午（无足够时间处理问题）
3. 预留足够时间处理异常

### 发布人员配置
- 执行人：负责发布操作
- 监控人：负责监控指标
- 产品：负责功能验证
- 运维：负责异常处理

### 发布观察期
- 发布后观察30分钟
- 监控关键指标（错误率、响应时间）
- 验证核心功能

### 回滚准备
- 回滚脚本预先准备并测试
- 数据备份预先完成
- 回滚决策流程明确
