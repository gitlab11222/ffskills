# 复盘规范

## 复盘时机

### 复盘类型
| 类型 | 时间节点 | 复盘内容 | 参与人员 |
|------|----------|----------|----------|
| 流程复盘 | 上线后一周内 | 时间线、流程问题、团队协作 | 产品、研发、测试、运维 |
| 效果复盘 | 上线后一个月 | 业务指标、用户反馈、数据分析 | 产品、运营、商务 |

## 复盘准备

### 数据准备
| 数据类型 | 内容 | 数据来源 |
|----------|------|----------|
| 时间线数据 | 各节点实际时间 | 项目记录 |
| 工时数据 | 各角色实际工时 | 团队记录 |
| Bug数据 | Bug数量、级别、分布 | TAPD/测试报告 |
| 业务数据 | 业务指标达成情况 | 数据统计 |
| 用户反馈 | 用户评价、投诉、建议 | 客服/调研 |

### 文档准备
- PRD文档
- 测试报告
- 发布记录
- Bug清单
- 用户反馈清单

## 复盘流程

### 流程复盘步骤
```
时间线回顾 → 问题识别 → 根因分析 → 改进建议 → 迭代计划
```

### 效果复盘步骤
```
目标回顾 → 数据分析 → 用户反馈 → 效果评估 → 迭代规划
```

### 4.2 五问法分析示例
```
问题: 开发延期3天

第一问: 为什么开发延期？
答: 后端接口开发耗时超出预期。

第二问: 为什么接口开发超预期？
答: 订单状态流转逻辑复杂，需要多次重构。

第三问: 为什么状态流转逻辑复杂？
答: 需求文档中状态定义不清晰，开发过程中多次调整。

第四问: 为什么状态定义不清晰？
答: 原型评审时未充分讨论状态流转细节。

第五问: 为什么评审时未充分讨论？
答: 评审时间紧张，未深入讨论边界场景。

结论: 评审流程需要优化，确保边界场景充分讨论。
```

## 复盘会议规范

### 会议准备
| 准备项 | 内容 | 负责人 |
|--------|------|--------|
| 数据准备 | 时间线、工时、Bug数据 | 项目经理 |
| 文档准备 | PRD、测试报告、反馈清单 | 项目经理 |
| 问题梳理 | 主要问题清单 | 全员 |
| 建议草拟 | 初步改进建议 | 全员 |

### 会议流程
| 步骤 | 内容 | 时长 | 主持人 |
|------|------|------|--------|
| 开场 | 说明复盘目的和流程 | 5分钟 | 项目经理 |
| 时间线回顾 | 回顾项目关键节点 | 15分钟 | 项目经理 |
| 问题分析 | 分析主要问题和根因 | 30分钟 | 全员讨论 |
| 经验总结 | 总结做得好和需改进 | 15分钟 | 全员讨论 |
| 改进建议 | 讨论改进措施 | 20分钟 | 全员讨论 |
| 下期规划 | 规划下期重点 | 10分钟 | 产品/项目经理 |
| 结束 | 确认待办和后续跟踪 | 5分钟 | 项目经理 |

### 会议输出
- 复盘报告（文档）
- 改进措施（待办清单）
- 迭代计划（排期）

## 最佳实践

### 复盘定期化
- 流程复盘：上线后一周内
- 效果复盘：上线后一个月

### 复盘数据化
- 所有结论有数据支撑
- 避免主观判断

### 改进落地化
- 改进措施有负责人和时间
- 定期跟踪改进落地情况

### 知识沉淀化
- 复盘报告存档
- 经验教训纳入规范
